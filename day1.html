H1 SwiGLU Kernel (SiLU Activation Path)
Critical ~5-6 hrs

Fuse SiLU(gate) * up into one kernel — the SiLU activation path used by Qwen3. Eliminates 2 intermediate tensor writes to HBM per layer. This is ~50% of Qwen3 compute — the single highest-impact kernel. P6 builds the GELU path (GeGLU for Gemma) on top of this same kernel file — coordinate with P6 at kickoff on the shared kernel interface.

Recommended Steps
  1. Read forge_learning/module_4_kernel_math/01_gradient_derivation_swiglu.md — understand the product-rule backward
  2. Read forge_learning/module_5_kernel_implementation/01_swiglu_triton_kernel.md
  3. Coordinate with P6 at 10:30: agree on kernel interface — the activation parameter name, how it's passed to the Triton kernel (compile-time tl.constexpr), where the activation branch happens in fwd and bwd
  4. Write _swiglu_fwd_kernel: load gate & up chunks, compute SiLU(gate) * up in registers, write output
  5. Test forward against F.silu(gate) * up — must match at bf16 within rtol=1e-5
  6. Write _swiglu_bwd_kernel: dgate = dY * up * sigmoid(gate) * (1 + gate*(1-sigmoid(gate))), dup = dY * SiLU(gate)
  7. Write ForgeSwiGLUFunction(autograd.Function) — save gate for backward, recompute sigmoid (Liger's approach)
  8. Write ForgeSwiGLU(nn.Module) wrapper with activation="silu" as default parameter
  9. Write test_swiglu.py: gradcheck (float64), forward correctness (bf16/fp16), backward correctness, shape sweep — all for SiLU path
  10. Write bench_swiglu.py using benchmarks/harness.py
How to approach it
  • Forward first. Get _swiglu_fwd_kernel passing correctness before touching backward.
  • Save gate, recompute sigmoid in backward (Liger's approach) — saves VRAM vs saving sigmoid directly.
  • Start with BLOCK_SIZE=1024, add @triton.autotune with the same 22-config set from RMSNorm.
  • Shape sweep must include Qwen3's intermediate=11008: test at [4, 2048, 11008].
  • Design the kernel with an activation branch point so P6 can add the GELU path without restructuring. Use a tl.constexpr parameter that selects SiLU vs GELU at compile time (zero overhead).
  • If you finish SiLU early, help P6 with the GELU backward derivative.
Done When
  • gradcheck PASSES at float64 with shape [2, 8, 4096] (SiLU path)
  • Forward output matches F.silu(gate) * up within rtol=1e-5 at bf16 and fp16
  • Backward gradients match PyTorch autograd (SiLU derivative)
  • Benchmark shows ≥1.0× PyTorch speed (hard floor), target ≥1.3×
  • Kernel interface has activation parameter ready for P6 to add the GELU path
● P1: SwiGLU SiLU path lead (coordinates with P6 on shared kernel file)
H2 RoPE Kernel (Fused Q+K)
High ~4-5 hrs

Apply rotary positional embeddings to Q and K jointly in one kernel launch. Backward is just inverse rotation (negate sin). Unsloth fuses Q+K into one launch for ~2.3× speedup over separate application.

Recommended Steps
  1. Read forge_learning/module_4_kernel_math/02_gradient_derivation_rope.md
  2. Read forge_learning/module_5_kernel_implementation/02_rope_triton_kernel.md
  3. Write _rope_fwd_kernel with 2D grid (num_rows, 2) — program 0 handles Q, program 1 handles K
  4. Pre-compute cos/sin frequency tables: freqs = 1.0 / (base ** (arange(0, dim, 2) / dim))
  5. Write ForgeRoPEFunction — backward calls forward with negated sin
  6. Write ForgeRoPE(nn.Module) with configurable base parameter
  7. Write test_rope.py: gradcheck, correctness vs HF apply_rotary_pos_emb
  8. Write bench_rope.py
How to approach it
  • Unsloth is the reference, not Liger. Liger applies RoPE to Q and K separately (two launches). Unsloth fuses them.
  • Math: x_rot = x * cos + rotate_half(x) * sin. Backward = negate sin, same operation.
  • Head dimension: Qwen3 has head_dim=128. RoPE applies to first rotary_dim dimensions only — make configurable.
  • Make base a parameter (default 10000.0) — Gemma uses a different rotary base. Not a separate kernel.
  • Start by wrapping HF's existing RoPE in a test harness to get reference outputs.
Done When
  • gradcheck PASSES at float64
  • Output matches HF apply_rotary_pos_emb within rtol=1e-5
  • Fused Q+K confirmed faster than separate application
  • cos/sin table precomputed once, not per-call
● P2: RoPE kernel lead
H3 Cross-Entropy Standalone Kernel
High ~5-6 hrs

Chunked cross-entropy loss over vocab dimension using online softmax. Avoids materializing the full [batch×seq, vocab=151936] logit tensor. Primary win is memory, not speed.

Recommended Steps
  1. Read forge_learning/module_4_kernel_math/03_online_softmax_algorithm.md
  2. Read forge_learning/module_4_kernel_math/04_fused_cross_entropy_math.md
  3. Study chapter_fused_linear_ce.md — extract the CE-only computation logic
  4. Write _ce_fwd_kernel: process vocab in chunks, accumulate running max + sum-exp, compute loss
  5. Write _ce_bwd_kernel: dlogits = softmax(logits) - one_hot(target)
  6. Write ForgeCrossEntropyFunction and ForgeCrossEntropy module
  7. Add label_smoothing parameter
  8. Write test_ce.py: gradcheck, correctness vs F.cross_entropy
  9. Write bench_ce.py — measure both speed AND memory
How to approach it
  • You already have Fused Linear+CE done. The standalone CE reuses the CE computation logic but takes pre-computed logits. Extract from the existing kernel as a starting point.
  • Online softmax is the key: one pass, running max + running sum-exp. Well-documented in Module 4.
  • Chunk size: start with CHUNK_SIZE=4096. Process vocab in chunks, combine via log-sum-exp trick.
  • fp32 accumulation is critical — all softmax reductions in fp32 even if inputs are bf16.
  • Memory target: ≤50% of F.cross_entropy. This is the primary win, not speed.
Done When
  • gradcheck PASSES at float64
  • Loss matches F.cross_entropy within rtol=1e-5
  • Peak memory ≤50% of F.cross_entropy at vocab=151936
  • Label smoothing produces correct results
● P3: Cross-Entropy kernel lead
H4 LoRA MLP Fused Kernel
Structural ~4-5 hrs

Pre-merge base weight W with LoRA A@B: compute (W + A@B) @ x as one matmul instead of two. Backward computes dA and dB only (W is frozen). A structural optimization — 1 matmul vs 2.

Recommended Steps
  1. Read forge_learning/module_4_kernel_math/05_lora_matmul_optimization.md
  2. Read forge_learning/module_5_kernel_implementation/04_lora_fused_kernels.md
  3. Write weight-merging: W_merged = W + A @ B where A is [hidden, rank], B is [rank, intermediate]
  4. Write ForgeLoRAMLPFunction(autograd.Function): forward uses F.linear(x, W_merged); backward computes dA, dB
  5. Write ForgeLoRAMLP(nn.Module) wrapper
  6. Write test_lora_mlp.py: output matches (W + A@B) @ x, gradcheck on A and B, verify W has no gradient
  7. Write bench_lora_mlp.py
How to approach it
  • Start simple: this is NOT a Triton kernel initially. Pre-merge weights, call one F.linear. That alone halves matmul count.
  • LoRA rank is typically 16-64. A is [4096, 16], B is [16, 11008]. Merge A@B is one-time cost per step.
  • Three projections in Qwen3 MLP: gate_proj, up_proj, down_proj. Each gets its own LoRA adapter.
  • Verify: W.requires_grad must be False. Only A and B get gradients.
  • If the merge itself is expensive, fuse it into a Triton matmul later — but start with pure torch.
Done When
  • Output matches (W + A@B) @ x exactly
  • gradcheck on A and B PASSES; W has no gradient
  • Measurably faster than unfused (W@x + A@(B@x))
● P4: LoRA MLP kernel lead
H5 LoRA QKV Fused Kernel
Structural ~6-7 hrs

Same as LoRA MLP but for attention projections, handling GQA asymmetry: Q is [4096, 4096] (32 heads), K/V are [4096, 512] (4 heads each). One fused operation for all three projections. Most complex kernel — may spill into Day 2.

Recommended Steps
  1. Read forge_learning/module_5_kernel_implementation/04_lora_fused_kernels.md — QKV section
  2. Concatenate W_q, W_k, W_v → one matrix [4096, 5120] (4096+512+512)
  3. Merge LoRA: W_merged = concat(W_q + Aq@Bq, W_k + Ak@Bk, W_v + Av@Bv)
  4. Forward: one matmul F.linear(x, W_merged), split output into Q, K, V
  5. Write ForgeLoRAQKVFunction — backward computes dA, dB for each of Q, K, V adapters
  6. Write tests: verify Q shape [batch, seq, 32, 128], K/V shape [batch, seq, 4, 128]
  7. Verify gradients flow to correct LoRA adapters (not cross-contaminated)
How to approach it
  • GQA is the tricky part. Q has 32 heads × 128 dim = 4096, K/V have 4 heads × 128 dim = 512. Different output sizes.
  • Start simple: three separate F.linear calls with pre-merged weights. Verify correctness. Then fuse by concatenating.
  • Unsloth's approach: concatenate W_q, W_k, W_v → [4096, 5120], one matmul, split output.
  • Test GQA explicitly: after fused forward, split output and verify Q[:,:,:32,:] matches reference Q.
  • If running late by 4pm, ship the "three separate pre-merged" version. It's still a win (3 merged matmuls vs 6 unfused). Optimize to single matmul later.
Done When
  • Q/K/V output shapes match GQA spec (32 heads Q, 4 heads K/V)
  • Fused output matches three separate projections
  • gradcheck on all 6 LoRA matrices (Aq, Bq, Ak, Bk, Av, Bv)
  • Measurably faster than unfused (1 or 3 matmuls vs 6)
● P5: LoRA QKV kernel lead
H6 GeGLU Kernel (GELU Activation Path)
Critical ~5-6 hrs

Build the GELU activation path for Gemma's GeGLU MLP: GELU(gate) * up instead of SiLU(gate) * up. Works on the same kernel file as P1 (H1). P1 writes the SiLU path; you write the GELU path. The forward is a simple activation swap, but the backward derivative is completely different: GELU's derivative involves the error function (erf), not sigmoid. This needs its own gradcheck, numerical stability testing, and benchmarking.

Recommended Steps
  1. Read forge_learning/module_4_kernel_math/01_gradient_derivation_swiglu.md — understand the product-rule backward structure (same for both activations)
  2. Coordinate with P1 at 10:30: agree on kernel interface — the activation parameter, how it branches in the Triton kernel (tl.constexpr), shared autograd.Function structure
  3. Write the GELU forward path in the shared Triton kernel: GELU(gate) * up when activation == "gelu"
  4. Write the GELU backward path — this is the hard part. The derivative of GELU is: d_gelu = 0.5 * (1 + erf(x/sqrt(2))) + x * exp(-x^2/2) / sqrt(2*pi). So: dgate = dY * up * d_gelu(gate), dup = dY * GELU(gate)
  5. Write test_geglu.py: gradcheck at float64 (critical — erf derivative is tricky), forward correctness vs F.gelu(gate) * up at bf16/fp16, backward correctness
  6. Test numerical stability: GELU's erf and exp(-x^2/2) can behave differently at extreme values vs SiLU's sigmoid. Test at large input magnitudes.
  7. Write bench_geglu.py: benchmark at Gemma shapes (hidden=2048, intermediate=16384)
  8. Verify: ForgeSwiGLU(dim, intermediate, activation="gelu") produces correct output end-to-end
How to approach it
  • Wait for P1 to land the kernel structure first (forward kernel + autograd.Function skeleton). P1 should have the SiLU forward working by ~12:00. Then you add the GELU branch.
  • While waiting for P1's skeleton, write your tests first (test-driven): test_geglu.py that tests ForgeSwiGLU(..., activation="gelu"). When the kernel is ready, run the tests.
  • The backward is where you'll spend most time. SiLU backward uses sigmoid (simple, numerically stable). GELU backward uses erf + exp(-x^2/2) (more complex, potential numerical issues). Derive carefully, verify with gradcheck at float64.
  • Triton has tl.math.erf and tl.math.exp. Use fp32 accumulation for the erf computation even if inputs are bf16.
  • If P1 is behind schedule, help them finish the shared kernel structure first — you can't add GELU until the SiLU skeleton exists.
Done When
  • gradcheck PASSES at float64 with shape [2, 8, 4096] (GELU path)
  • Forward output matches F.gelu(gate) * up within rtol=1e-5 at bf16 and fp16
  • Backward gradients match PyTorch autograd (GELU/erf derivative)
  • Numerical stability verified at large input magnitudes (no NaN/Inf from erf or exp)
  • Benchmark at Gemma shapes shows ≥1.0× PyTorch speed
● P6: GeGLU path lead (coordinates with P1 on shared kernel file)

Infrastructure ~5-6 hrs

Build the forge.patch(model) / forge.unpatch(model) machinery and the Qwen3 + Gemma module mapping files. Also handles the two trivial Gemma kernel parameter changes (RMSNorm offset, RoPE base), traces the Gemma module tree, and researches FSDP2 compatibility. Recommended Steps
  1. Study integration_gpt2large/integration_test.ipynb — understand the monkey-patching pattern and closure factory. Then read Design Details → Wiring Guide before writing any code. The Wiring Guide shows the exact closure factory pattern, the _FORWARD_MAKERS dispatch table, and _replace_forward. H8’s job is to build the outer shell (patch.py, qwen3.py, gemma.py); the Wiring Guide shows how the inner dispatch works.
  2. Create forge/forge/patching/patch.py — use this skeleton: 
    from .qwen3 import QWEN3_MAPPING
from .gemma import GEMMA_MAPPING

ARCH_MAPPINGS = {
    "qwen2":  QWEN3_MAPPING,   # Qwen2.5 / Qwen3 family
    "gemma2": GEMMA_MAPPING,
}

def patch(model, kernels=None):
    model_type = model.config.model_type   # e.g. "qwen2", "gemma2"
    mapping = ARCH_MAPPINGS.get(model_type)
    if mapping is None:
        raise ValueError(f"No Forge mapping for model_type={model_type!r}")
    originals = {}
    for _, module in model.named_modules():
        class_name = type(module).__name__
        if class_name in mapping:
            kernel_name, config = mapping[class_name]
            if kernels is None or kernel_name in kernels:
                _replace_forward(module, kernel_name, config, originals)
    model._forge_originals = originals

def unpatch(model):
    for mod_id, orig_fwd in model._forge_originals.items():
        ...  # restore original forward on each module
    del model._forge_originals

  11. Write test_fsdp2_smoke.py — use this skeleton: 
    # run with: torchrun --nproc-per-node=2 test_fsdp2_smoke.py
import torch, torch.distributed as dist, functools
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy
from forge.kernels.rmsnorm import ForgeRMSNorm
import torch.nn as nn

dist.init_process_group("nccl")
rank = dist.get_rank()
torch.cuda.set_device(rank)

class TinyBlock(nn.Module):
    def __init__(self):
        super().__init__()
        self.norm = ForgeRMSNorm(4096)
        self.linear = nn.Linear(4096, 4096)
    def forward(self, x):
        return self.linear(self.norm(x))

# Single-GPU reference (rank 0 only)
if rank == 0:
    ref = TinyBlock().cuda().bfloat16()
    x_ref = torch.randn(4, 128, 4096, dtype=torch.bfloat16, device="cuda")
    ref(x_ref).sum().backward()
    ref_grad = ref.norm.weight.grad.clone()

# FSDP-wrapped run
model = nn.Sequential(TinyBlock()).cuda().bfloat16()
model = FSDP(model, auto_wrap_policy=functools.partial(
    transformer_auto_wrap_policy, transformer_layer_cls={TinyBlock}
))
x = torch.randn(4, 128, 4096, dtype=torch.bfloat16, device="cuda")
out = model(x); loss = out.sum(); loss.backward()

# Four checks
assert not out.isnan().any(), f"FAIL rank {rank}: NaN in output"
assert not loss.isnan(),      f"FAIL rank {rank}: NaN in loss"
param_bytes = sum(p.numel() * p.element_size() for p in model.parameters())
print(f"rank {rank}: param bytes held = {param_bytes:,}")  # check VRAM sharded
if rank == 0:
    fsdp_grad = model[0].norm.weight.grad
    max_diff = (fsdp_grad - ref_grad).abs().max().item()
    assert max_diff < 1e-3, f"FAIL: grad mismatch max_diff={max_diff:.6f}"
    print(f"PASS: loss={loss.item():.4f}, grad_max_diff={max_diff:.6f}")

dist.destroy_process_group()
