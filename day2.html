H9 Port Day 1 Kernels into Package
Blocking ~2 hrs

Port the 5 new kernels from Day 1 into the forge package and register them. Since the package scaffold already exists from H7, this is pure porting work — ~2 hours instead of the original 2.5+.

Steps
  1. Collect all 5 Day 1 kernel files (SwiGLU, RoPE, CE, LoRA MLP, LoRA QKV)
  2. Port each into forge/forge/kernels/: fix imports, add @register_kernel decorator
  3. If any Day 1 kernel was integrated late on Day 1, just clean up the port
  4. Verify: forge.kernels.list() returns all 9 kernel names
  5. Run all kernel tests from within the package to verify imports work
Done When
  • forge.kernels.list() returns all 9 kernel names
  • Each kernel importable from the package
  • All kernel tests pass from within the package
● P7 (built the scaffold on Day 1 — knows the structure)
H10 forge.patch / forge.unpatch (Qwen3)
Core ~3 hrs

Wire up the patching infrastructure (built on Day 1 by P8) to real kernels on Qwen3. The patch.py code and qwen3.py mapping already exist — this is testing, debugging, and getting it to produce valid logits.

Steps
  1. Wait for H9 to register at least the kernels needed for Qwen3 patching
  2. Wire qwen3.py mapping to actual registered kernels (connect module types to real Forge modules)
  3. Test: forge.patch(model) on Qwen2.5-0.5B, run forward pass, verify valid logits
  4. Test: forge.unpatch(model) restores exact original output
  5. Test: compose with get_peft_model() for LoRA (patch before LoRA)
  6. Debug any issues — enable kernels one at a time: forge.patch(model, kernels=["rmsnorm"])
How to approach it
  • Read Design Details → Wiring Guide before writing any code. It has the exact closure factories, dispatch table, and incremental test script.
  • You wrote the patching code on Day 1. You know the internals. This is wiring + debugging, not design.
  • Start with a single kernel (RMSNorm) to verify the pattern end-to-end, then enable all.
  • If a forward pass produces NaN after patching, enable one kernel at a time to isolate the problem kernel.
  • Must be done by 13:30 (CP-H5a) to unblock the training demo.
Done When
  • forge.patch(model) on Qwen2.5-0.5B produces valid logits (no NaN, correct shape)
  • forge.unpatch(model) restores original output exactly
  • Patched model composes with get_peft_model() — LoRA + Forge kernels together
  • Max logit diff between patched and unpatched < 1.0
● P8 (wrote the patching infra on Day 1)
H11 forge.patch (Gemma) + Gemma Training
Core ~4 hrs

Wire up Gemma patching using the infra from Day 1. The gemma.py mapping file exists — connect it to real kernels with Gemma-specific configs (RMSNorm offset=1.0, GeGLU, Gemma RoPE base). Write and run the Gemma training demo.

Steps
  1. Wire gemma.py mapping to registered kernels with Gemma-specific config overrides
  2. Test: forge.patch(model) on gemma-2-2b, forward pass produces valid logits
  3. Test: forge.unpatch(model) restores original output exactly
  4. Test: compose with get_peft_model() for LoRA
  5. Write examples/sft_gemma_lora.py: LoRA SFT on gemma-2-2b with Forge kernels
  6. Run 100-step training: verify loss decreases, no NaN, all Gemma-variant kernels active
  7. Write forge/tests/test_integration_gemma.py: patched vs unpatched logit comparison
How to approach it
  • Read Design Details → Wiring Guide before writing any code. It has the closure factory pattern and the exact incremental test script. The only difference from Qwen3 is the module class names and config dict values.
  • You built the Gemma kernel variants on Day 1 — you know exactly which params each kernel needs.
  • Reuse P8's patching infrastructure. gemma.py is the same structure as qwen3.py, just different module names and config overrides.
  • The training script is almost identical to Qwen3's — swap model name and adjust target_modules if Gemma names projections differently.
  • Must have patching working by 14:00 (CP-H5b) to stay on track for the dual-model demo.
Done When
  • forge.patch(model) on gemma-2-2b produces valid logits (no NaN, correct shape)
  • Gemma-specific kernel params applied correctly (offset=1.0, gelu, Gemma RoPE base)
  • examples/sft_gemma_lora.py runs 100 steps, loss decreases, no NaN
  • Integration test: patched vs unpatched max logit diff < 1.0, cosine sim > 0.999
● P6 (built GeGLU kernel on Day 1 — knows the Gemma-specific kernel params)
H12 Training Pipeline + Demo Scripts (Qwen3 + Gemma)
Core ~4-5 hrs

Write end-to-end LoRA SFT training scripts for both Qwen3 and Gemma. These are the hackathon demo. First run without Forge kernels to verify baseline, then add forge.patch.

Steps
  1. Write examples/sft_qwen3_lora.py: load Qwen2.5-0.5B, forge.patch, apply LoRA, custom training loop — see Design Details → Training Loop for the exact 40-line script
  2. Use a tiny dataset (synthetic or HuggingFaceTB/smollm-corpus) — 1000 samples max
  3. Step A (Qwen3): Run WITHOUT forge.patch — verify vanilla training works (loss decreases)
  4. Step B (Qwen3): Add forge.patch(model) before LoRA — verify loss still decreases
  5. Step C (Gemma): Write examples/sft_gemma_lora.py — same pattern with google/gemma-2-2b (coordinate with H11)
  6. Step D (Gemma): Add forge.patch(model) — verify Gemma training also works
  7. If loss diverges on either model: forge.patch(model, kernels=["rmsnorm"]) → enable one at a time to isolate
  8. Print per-step metrics: loss, learning rate, VRAM usage for both models
  9. Write examples/sft_qwen3_full_ft.py (full fine-tuning, if time permits)
How to approach it
  • Read Design Details → Training Loop before writing any code. It has the full script, dataset setup, the critical forge.patch → get_peft_model ordering, and locked decisions.
  • Start the baseline (Step A) immediately at 10:15. Doesn't need forge.patch. Get vanilla training working while H9/H10 finish.
  • Use Qwen2.5-0.5B and gemma-2-2b (small models that fit on any GPU).
  • Coordinate with H11 (Gemma patching) for the Gemma demo script.
  • 100 steps is enough for the demo. If patched loss shows NaN: enable kernels one at a time.
Done When
  • Qwen3: Forge-patched LoRA SFT runs 100 steps, loss decreases, no NaN
  • Gemma: Forge-patched LoRA SFT runs 100 steps, loss decreases, no NaN
  • All 9 Forge kernels confirmed active during training on both models
● P1 (wrote SwiGLU on Day 1 — knows the kernel internals for debugging)

H15 FSDP2 Smoke Test (2-GPU)
Core ~3-4 hrs

Run the FSDP2 smoke test script (written by P8 on Day 1) on a 2-GPU node. Verify that our autograd.Function pattern works under FSDP2 sharding. This de-risks the entire Phase 3 of the V1 plan and gives the team a head start on multi-GPU work starting Monday. See the FSDP2 Smoke Test deep-dive for the full P8 working doc.

Steps
  1. Set up 2-GPU node (should be provisioned Friday)
  2. Install forge package on the 2-GPU node
  3. Run P8's test_fsdp2_smoke.py: torchrun --nproc-per-node=2 test_fsdp2_smoke.py
  4. Compare gradients between 2-GPU FSDP2 run and single-GPU reference
  5. If it fails: investigate ctx.save_for_backward behavior with sharded weights. Document the failure mode.
  6. If it passes: try with more kernels (not just RMSNorm). Test SwiGLU and RoPE under FSDP2.
  7. Document: PASS/FAIL, pattern requirements, Phase 3 playbook with specific recommendations
  8. If FSDP2 passes early, try LoRA + forge.patch + FSDP2 composition (the Phase 3 holy grail)
How to approach it
  • The script is already written (by P8 on Day 1). Your job is to run it, debug failures, and document findings.
  • Start in the morning while others are porting/wiring. The 2-GPU node is independent of the main development flow.
  • If the smoke test fails, that's still a valuable finding — document exactly what breaks and what pattern change is needed. This saves days of debugging in Phase 3.
  • Key question to answer: does ctx.save_for_backward(weight) work when FSDP2 has discarded the full weight after forward?
Done When
  • torchrun --nproc-per-node=2 test_fsdp2_smoke.py — result documented (PASS or FAIL with diagnosis)
  • Gradients compared to single-GPU reference
  • Phase 3 playbook written: what works, what doesn't, recommended patterns
  • If PASS: at least 2 additional kernels verified under FSDP2

Sunday Evening — Retrospective & Next 4 Weeks Planning

Reserved block: 2 hours, 19:00–21:00 (after dinner). The hackathon does not end at the demo. This block converts the day's work into a tracked plan for the next 4 weeks. Every engineer leaves with a clear owner-and-deliverable list. No retro = scattered followup work + lost momentum.

Retro Two-hour agenda — reserved on the schedule
19:00 – 21:00 Everyone (8 people)

Tight agenda. Timekeeper enforces the clock so we don't drift. Notes captured in a shared doc as we go.

  1. Round-robin status (30 min, ~3 min per person). Each engineer shares: what shipped (passed all 7 done-gates), what's marked "needs follow-up" and why, what surprised them. No discussion yet — just listen.
  2. What slipped and why (20 min). Honest postmortem on anything dropped from the scope ladder. Was it a missing prerequisite? A kernel bug that ate hours? An AI hallucination that wasn't caught? Capture the systemic learnings, not the blame.
  3. Smoke test verdict deep-dive (15 min). P8 walks through the H15 outcome (A–F from the smoke-test deep dive). What does it mean for Phase 3? Pattern change required? If yes, what exactly?
  4. Recalibrate the V1 plan (45 min). Pull up FORGE_V1_TARGETS.md. Walk through the four checkpoints and adjust based on hackathon outcomes:
    • CP2 (June 14): Single-GPU LoRA + Full FT on Qwen3 + Gemma. What's already done after hackathon. What remains.
    • CP3 (June 28): 4-GPU FSDP2 training. Plan revised by smoke-test verdict.
    • CP4 / V1 (July 15): Benchmarks, stress tests, ship. Date confirmed or moved.
  5. Followup capture (10 min). Open a GitHub issue for every "needs follow-up" kernel from Step 1. Assign owners. Tag with milestone (CP2/CP3/CP4).
Done When (CP-H8 by 21:00)
  • Every "needs follow-up" item has a tracked issue with an owner
  • The V1 plan is updated with hackathon-adjusted dates and scope
  • Every engineer knows what they're working on Monday morning
  • Smoke test outcome is documented and Phase 3 plan reflects it
  • Shared notes doc captures the systemic learnings (for the next sprint)
Anti-goals
  • Not a celebration. Celebration happens during dinner (18:00). This block is work.

  • Not optional. All 8 engineers attend. The plan they leave with directly affects whether V1 ships July 15.
  • Don't run past 21:00. People are tired. Decisions degrade. End on time.
