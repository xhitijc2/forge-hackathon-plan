Planning — Dependencies, Risks & Stretch

The "what could go wrong / what if we have extra time" reference. Day 1 / Day 2 dependency map, nine risks with mitigations, two stretch goals if both demos ship and FSDP2 passes before 4:30pm.

Dependency Map

Day 1 runs 8 tracks with one soft dependency: H6 (GeGLU) needs H1's (SwiGLU) kernel skeleton before adding the GELU path. Everything else is fully parallel. Day 2 has a short critical path: H9 (port kernels) unblocks H10/H11 (patching) which unblocks H12 (training). But the bottleneck is ~2 hours, not 2.5+ — because the scaffold was built on Day 1.

Risks & Contingency

High LoRA QKV fused kernel (H5) takes longer than a day

GQA asymmetry gives this kernel the most moving parts. May not complete forward+backward+tests in one day.

✓ Mitigation: Ship the "three separate pre-merged matmuls" version by 4pm. It's still a win (3 merged vs 6 unfused). Fuse into single matmul post-hackathon.

Medium P1/P6 shared kernel coordination causes merge conflicts or blocking

P1 (SiLU) and P6 (GELU) work on the same kernel file. If P1 is behind schedule, P6 is blocked. If they don't agree on the interface, they waste time refactoring.

✓ Mitigation: Lock the interface at 10:30am (parameter name, constexpr branch point, autograd.Function skeleton). P6 writes tests first while waiting for P1's skeleton. If P1 is behind by 13:00, P6 helps P1 finish the shared structure before starting GELU.

High forge.patch(model) breaks with LoRA composition

Patching + LoRA together may produce unexpected behavior (wrong module wrapping order, missing gradients, forward hook conflicts).

✓ Mitigation: Test patch-only first (no LoRA). Then add LoRA. If composition fails, demo with patch-only forward pass. Debug composition post-hackathon.

Medium A backward kernel has a numerical bug that only shows during training

Kernel passes gradcheck at float64 but produces NaN at bf16 during actual training (different scale).

✓ Mitigation: Training script (H12) uses forge.patch(model, kernels=["rmsnorm"]) to enable one kernel at a time. Isolate the offending kernel in <30 min. Disable it for the demo and fix post-hackathon.

Medium Day 1 infra (H7/H8) is untested with new kernels

Package scaffold and patching infra are built on Day 1 against existing kernels. New Day 1 kernels may have different interfaces or import patterns that break the integration on Day 2.

✓ Mitigation: P7 integrates any early-finishing Day 1 kernels (H4 LoRA MLP likely finishes first) to catch interface issues on Day 1 itself. Kernel writers follow the existing RMSNorm pattern, minimizing surprises.

Medium Gemma architecture has unexpected module differences

Gemma's HF implementation may name modules differently or have subtle architectural differences beyond RMSNorm offset / GeGLU / RoPE base.

✓ Mitigation: P6 traces Gemma modules on Day 1 (before patching starts) and shares with P8. If major differences are found, scope Gemma patching to the kernels that map cleanly.

Medium Gemma training eats time that could stabilize Qwen3

Covering two models in 2 days is ambitious. Gemma debugging might steal hours from stabilizing the Qwen3 pipeline.

✓ Mitigation: H11 (Gemma) and H10 (Qwen3) are assigned to different people (P6 and P8). If Gemma is behind at 3pm, P6 pivots to help with Qwen3 demo. Ship one model perfectly over two models buggily.

Medium FSDP2 smoke test fails

ctx.save_for_backward(weight) may not work correctly when FSDP2 has sharded and discarded the full weight after forward.

✓ Mitigation: A documented failure is still a win — it tells us exactly what pattern change is needed for Phase 3. P3 documents the failure mode and writes the fix recommendation. No time wasted guessing in Phase 3. See the smoke-test deep dive for the full P8 playbook.

Low GPU provisioning delays

Cloud GPU instances take time to spin up. Time lost at start of Day 1.

✓ Mitigation: Provision instances on Friday evening (May 22). Have environments ready with PyTorch, Triton, transformers pre-installed. Include one 2-GPU node for the FSDP2 smoke test.

Stretch Goals

If both demos work and FSDP2 passes by 4:30pm Sunday. These are bonus — don't sacrifice core targets for them.

S1 500-Step Convergence Test
Stretch ~1-2 hrs

Run 500 steps with patched model and 500 steps with unpatched model (same seed, same data) on both Qwen3 and Gemma. Compare loss curves. This is the "0% accuracy loss" verification.

Done When
  • Loss curves plotted: patched vs unpatched over 500 steps (both models)
  • Max deviation within statistical noise at each checkpoint
S2 torch.compile Compatibility Check
Stretch ~30-60 min

Quick test: does torch.compile(model) work after forge.patch(model)? If yes, that's a free V1 feature. If no, document what breaks for post-hackathon fix.

Done When
  • Result documented: COMPATIBLE or INCOMPATIBLE with specific error
  • If compatible: forward + backward benchmarked with compile
