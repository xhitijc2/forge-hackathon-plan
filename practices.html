Engineering Practices

Two sources of truth for Day 1: the precise step-by-step workflow for building a kernel (with mandatory checkpoints at each step), and the dos and don'ts of AI-assisted kernel writing. Refer to this whenever in doubt about whether a kernel is ready to ship.

Good Practices — Kernel Build Workflow

5 steps. 5 step-level checkpoints. 7 done-gates. A kernel is done ONLY when all seven done-gates pass — no exceptions, no partial credit. If a gate fails, mark the kernel as "needs follow-up" and move on. Do not pretend a kernel is done when it isn't.

Step 1 Read the math — NO CODING YET
~30 min No code

Read three things in this exact order before writing a single line of code. Skipping this step is the #1 cause of multi-hour debug sessions later.

  1. Your forge_learning/module_4_kernel_math/* chapter — the gradient derivation. Read in full, including any worked examples.
  2. Your forge_learning/module_5_kernel_implementation/* chapter — the implementation prompt and the specific shape/dtype constraints.
  3. The Liger or Unsloth source file for the same kernel — line by line. Open the file in a tab and read every line. Compare against the math.
Checkpoint 1 — gate to Step 2 (all 4 required)
  • You can write the forward formula on paper without looking
  • You can write the backward formula on paper (both dL/dinput and dL/dweight if applicable)
  • You can list exactly which tensors get saved in ctx.save_for_backward
  • You can list the shape and dtype of every input and output tensor
Step 2 Forward only, with a PyTorch reference
~60-90 min Forward path

Build forward first. Backward returns None placeholders. Get forward correctness before touching backward — this is non-negotiable.

  1. Write a pure-PyTorch reference function first: def ref_forward(x, w): return .... This is your oracle.
  2. Write the Triton forward kernel (@triton.jit function).
  3. Write ForgeXxxFunction(autograd.Function) with forward wired up and backward returning None-filled tuple.
  4. Write test_forward_correctness: torch.allclose(forge_forward(x, w), ref_forward(x, w)) at 4+ shapes including Qwen3 shapes (hidden=4096, intermediate=11008).
  5. Test at bf16, fp16, and fp32 dtypes.
Checkpoint 2 — gate to Step 3 (all required)
  • test_forward_correctness PASSES at fp32 with rtol=1e-5
  • test_forward_correctness PASSES at bf16 with rtol=1e-2 at all 4+ shapes
  • No NaN or Inf in outputs at any tested shape
  • Do NOT proceed to backward until forward is bulletproof. A broken forward will make gradcheck failures impossible to diagnose.
Step 3 Backward kernel + gradcheck
~60-90 min Backward path

Now wire up backward. gradcheck at fp64 is the gold standard for backward correctness — if it passes, the math is right.

  1. Write the Triton backward kernel using the saved tensors from Step 2.
  2. Wire it into ForgeXxxFunction.backward.
  3. Run torch.autograd.gradcheck(ForgeXxxFunction.apply, inputs_fp64) with default tolerances. fp64 inputs eliminate precision noise — pure math check.
  4. Run test_backward_correctness: compare .grad from your Function vs PyTorch autograd on the reference, at bf16 and fp16.
If gradcheck fails
  • "max relative error" > 1e-3 at fp64: math bug in your backward kernel. Re-derive on paper. Check sign errors, missed terms, wrong broadcasting.
  • fp64 PASSES but bf16 diverges: precision bug, not math bug. You're missing tl.float32 accumulation in a reduction. fp64 hides this; bf16 surfaces it.
  • Forward affects backward gradient: if you tweaked Step 2 to fix gradcheck, you broke forward. Re-verify Step 2 first.
Checkpoint 3 — gate to Step 4 (all required)
  • torch.autograd.gradcheck(...) PASSES at fp64 with default tolerances
  • test_backward_correctness PASSES at bf16 with rtol=1e-2
  • No NaN in any gradient at any tested shape
  • Gradients are returned for the correct number of inputs (PyTorch will silently error otherwise)
Step 4 Performance + benchmarks
~60 min Benchmarks

Correctness alone is not enough — a correct kernel slower than PyTorch is a net regression. Use the shared harness; don't roll your own timing.

  1. Wrap the autograd.Function in an nn.Module.
  2. Write benchmarks/bench_<name>.py using benchmarks/harness.py:
    • benchmark_cuda() for timing (CUDA events, NOT time.time())
    • measure_memory() for peak VRAM
    • L2 cache flush between runs
    • 30 runs minimum, IQR filtering (per rules.txt)
  3. Benchmark against: PyTorch baseline, Liger equivalent, Unsloth equivalent.
  4. Test at Qwen3 shapes: seq_len ∈ {512, 1024, 2048, 4096}, batch ∈ {1, 4}, dtype bf16.
  5. Append one CSV row per shape to benchmarks/results/cp1_kernel_benchmarks.csv.
Checkpoint 4 — gate to Step 5 (all required)
  • Speed (fwd+bwd combined): ≥ 1.0× PyTorch at every tested shape (hard floor — no regression)
  • Peak memory: ≤ PyTorch at every tested shape (hard floor — no regression)
  • CSV row exists for every (kernel, shape, dtype) combination
  • If your kernel is critical (SwiGLU, RMSNorm, CE, Fused Linear+CE): meets the per-kernel target from the V1 plan (e.g., SwiGLU ≥ 1.3×, RMSNorm ≥ 1.2×, CE memory ≤ 50%)
Step 5 Writeup
~30 min Documentation

Brief markdown writeup. Doesn't have to be polished — it has to exist. Future-you debugging this kernel in 3 weeks will thank present-you.

  1. Follow the structure in teaching_guideline.md.
  2. Include: forward formula, backward formula, what's saved for backward, autotune config rationale, benchmark numbers.
  3. Commit alongside the kernel in chapter_<name>.md.
Checkpoint 5 — final gate
  • chapter_<name>.md exists at the repo root
  • Contains forward formula, backward formula, saved-tensors list, benchmark table
  • Anyone on the team can read it and rebuild your kernel from scratch

The 7 Done-Gates — All Must Pass

A kernel is done and dusted ONLY when every one of these 7 gates is green. If any gate is red, the kernel is NOT done. Mark it as "needs follow-up" in the team chat and move on — do not silently call it done.

# Gate How to check Pass threshold
G1 gradcheck at fp64 torch.autograd.gradcheck(Fn.apply, inputs_fp64) Returns True with default tolerances
G2 Forward correctness at bf16 + fp16 torch.allclose vs PyTorch reference rtol=1e-2 at bf16, 4+ shapes including Qwen3 (hidden=4096, intermediate=11008)
G3 Backward correctness at bf16 + fp16 torch.allclose(forge_grad, ref_grad) rtol=1e-2 at bf16, no NaN
G4 Speed ≥ 1.0× PyTorch benchmarks/harness.py CUDA event timing No regression fwd+bwd combined, at every tested shape
G5 Peak memory ≤ PyTorch torch.cuda.max_memory_allocated() No regression at every tested shape
G6 Benchmark CSV row exists Check benchmarks/results/cp1_kernel_benchmarks.csv One row per (kernel, shape, dtype)
G7 Code in expected file structure Files exist at expected paths forge/kernels/<name>.py + tests/test_<name>.py + benchmarks/bench_<name>.py + chapter_<name>.md
Critical-Kernel Stronger Floors (G4 raised)
  • SwiGLU / GeGLU: ≥ 1.3× PyTorch, within 15% of Liger
  • RMSNorm: ≥ 1.2× PyTorch
  • Cross-Entropy: peak memory ≤ 50% of F.cross_entropy
  • Fused Linear+CE: peak memory ≤ 40% of separate F.linear + F.cross_entropy
  • If a critical kernel passes G4 (≥ 1.0×) but misses the stronger target: shippable but flagged — done for Day 1, optimize post-hackathon.

Tips for Using AI

AI is doing a lot of the typing on Day 1. Used well, it's a 3-5× force multiplier. Used badly, it produces plausible-looking code that fails gradcheck in subtle ways and wastes hours. The mental model: AI types the boilerplate, you verify the math against the reference. Never the other way around.

Good ways to use AI

DO Anchor every prompt on a real reference

AI hallucinations drop dramatically when you give it concrete source to reason about. Always provide:

  • The Liger or Unsloth source file (paste the full file, not just a snippet)
  • Your module 4 gradient derivation (paste the relevant section)
  • A working kernel from your repo to mirror style/patterns (e.g., your RMSNorm kernel)
Example prompts that work
  • "Walk me through linkedin/Liger-Kernel/src/liger_kernel/ops/swiglu.py block by block — what does each tl.load correspond to in the math?"
  • "Given this gradient derivation [paste module 4 chapter], draft a Triton backward kernel. Use the same autotune config pattern as the RMSNorm kernel in our repo [paste]."
  • "My gradcheck is failing with max_relative_error=0.003. Here's my backward [paste full file]. Walk through line by line against the derivation [paste]. Where is the math wrong?"
  • "Generate edge-case tests for this Triton kernel: tiny shapes, non-aligned shapes, all-zeros input, all-NaN input."

Bad ways to use AI

DON'T No-context prompts and blind trust
Anti-patterns to avoid
  • "Write me a SwiGLU Triton kernel" — no context, no anchor reference. High hallucination risk. The kernel will look plausible and fail subtle correctness tests.
  • Trusting AI's first answer without running it. Always run gradcheck, not just allclose.
  • Asking AI for the gradient math. Use module 4 instead — that derivation is already verified by your team. AI is more likely to introduce errors here than help.
  • Copy-pasting code you don't understand. If you can't explain every line of the Triton kernel back to the team, you have a debt that will surface during integration on Day 2.
  • Pasting an error stack trace alone. Always include: the code, what you expected, what happened. "Fix this error: X" without context produces flailing.

Triton-specific AI failure modes (watch for these)

These are the bugs we've seen AI introduce most often when writing Triton kernels. Each one passes a casual code review and fails a careful one.

Failure mode Symptom How to catch it
Missing mask= on tl.load for partial tiles Silent out-of-bounds reads of garbage memory. gradcheck mysteriously fails at non-power-of-2 shapes. Test at non-aligned shapes (e.g., seq_len=1023, hidden=4097). If pass at 1024/4096 but fail at 1023/4097, this is it.
Wrong autotune configs Kernel runs and is correct, but is 3-5× slower than it should be. Benchmark against Liger. If you're >30% slower than Liger on similar hardware, suspect autotune. Copy our RMSNorm kernel's 22-config set as a starting point.
Backward kernel that LOOKS right but has subtle math errors gradcheck fails with small but non-trivial error (~1e-3 to 1e-2). gradcheck at fp64 is your safety net. Trust it over the kernel.
Missing fp32 accumulation in bf16 reductions gradcheck passes at fp64 but bf16 training NaNs after ~50 steps. Always cast to tl.float32 inside any reduction loop, even if inputs are bf16. Match the RMSNorm kernel's pattern.
Stale Triton API usage Code fails to compile or has deprecation warnings. AI training data may be 6-12 months old. Reference Triton's docs and your own working kernels, not just AI.
Wrong block-size assumptions Kernel works at small shapes, OOMs or hangs at Qwen3 shapes. Test at the actual Qwen3 shapes (hidden=4096, intermediate=11008) before declaring done.
Confused autograd.Function input/output count TypeError: backward() must return as many tensors as forward() input. Count inputs to forward; backward must return that many gradients (use None for non-differentiable inputs like the weight in some kernels).

The mental model

How to think about AI assistance
  • AI is a fast typist with imperfect math. Let it type. Verify the math yourself.
  • Anchor every prompt on a real reference. Liger/Unsloth source + module 4 derivation are your safety net.
  • Run gradcheck religiously. It's the only tool that reliably catches subtle math bugs that AI introduces.
  • If you can't explain a line of code, you don't own it. And on Day 2 when it breaks, you won't be able to fix it.
  • Pair on tricky kernels. Two people prompting AI together catches more issues than one person flying solo. P1+P6 on the shared SwiGLU/GeGLU file is the explicit example of this.
  • When AI confidently asserts something that contradicts the reference: trust the reference. Always.
