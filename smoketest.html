FSDP2 Smoke Test — H15 Deep Dive

P8's working doc for Day 2 afternoon. The minimum experiment that retires the single biggest architectural risk in V1: does the autograd.Function pattern survive FSDP2 sharding?

FSDP2 Smoke Test — H15 Deep Dive

Owner: P8  ·  Window: Day 2, 13:00–16:00 (3 hrs incl. cloud setup)  ·  Hardware: 2×A100 or 2×H100, on-demand. Blocker for: the entire V1 multi-GPU story. If this fails, Phase 3 needs a pattern redesign.

1

Purpose

The one question
  • Does our torch.autograd.Function pattern — specifically ctx.save_for_backward(weight, ...) — work correctly when FSDP2 has sharded the weight across GPUs?
  • Yes → every kernel using the same pattern is FSDP2-compatible. Phase 3 is unblocked.
  • No → we learn exactly which part breaks, and fix it once for all kernels before scaling out.
  • Not a benchmark. Not a convergence test. Not a full training run. The minimum experiment that retires the single biggest architectural risk in V1.
2

3

The Test Model

Minimal smoke-test model

import torch.nn as nn
from forge.kernels.rmsnorm import ForgeRMSNorm

class SmokeModel(nn.Module):
    def __init__(self, hidden=4096):
        super().__init__()
        self.norm = ForgeRMSNorm(hidden)        # ← the kernel under test
        self.proj = nn.Linear(hidden, hidden)   # ← standard layer for gradient flow

    def forward(self, x):
        return self.proj(self.norm(x))
Why this shape
  • ForgeRMSNorm has a learnable weight of shape [hidden] — this is the parameter FSDP2 will shard.
  • nn.Linear provides a known-good reference path for gradients (we trust PyTorch's nn.Linear under FSDP2).
  • Total params small enough to fit on one GPU for the reference run, large enough that sharding is non-trivial.
  • Why NOT a full transformer: attention + MLP + RoPE introduces 5+ failure modes. With the minimal model, any failure is definitively localized to RMSNorm's autograd.Function pattern.
4

The Procedure

Step 1 — Reference run (single GPU, rank 0, no FSDP2)

torch.manual_seed(42)
torch.cuda.manual_seed(42)
model = SmokeModel(hidden=4096).cuda()
x = torch.randn(4, 128, 4096, device='cuda', requires_grad=True)
y = model(x)
loss = y.sum()
loss.backward()

# Save references
ref_output       = y.detach().clone()
ref_input_grad   = x.grad.detach().clone()
ref_weight_grad  = model.norm.weight.grad.detach().clone()
torch.save({
    'output': ref_output,
    'input_grad': ref_input_grad,
    'weight_grad': ref_weight_grad,
}, 'reference.pt')

# Save state dict so the FSDP2 run starts from identical weights
torch.save(model.state_dict(), 'init_state.pt')

Step 2 — FSDP2 run (2 GPUs, torchrun --nproc-per-node=2)

import torch.distributed as dist
from torch.distributed.fsdp import fully_shard  # FSDP2 API

dist.init_process_group(backend='nccl')

torch.manual_seed(42)
model = SmokeModel(hidden=4096).cuda()
model.load_state_dict(torch.load('init_state.pt'))  # identical weights

# FSDP2 wrap
fully_shard(model.norm)   # shard the RMSNorm weight param
fully_shard(model.proj)   # shard the Linear

torch.manual_seed(42)
x = torch.randn(4, 128, 4096, device='cuda', requires_grad=True)
y = model(x)
loss = y.sum()
loss.backward()

fsdp_output      = y.detach().clone()
fsdp_input_grad  = x.grad.detach().clone()
# weight grad is sharded — all-gather it for comparison
fsdp_weight_grad = _all_gather_param_grad(model.norm.weight)

Step 3 — Compare

ref = torch.load('reference.pt')
assert torch.allclose(fsdp_output,      ref['output'],      rtol=1e-4, atol=1e-5)
assert torch.allclose(fsdp_input_grad,  ref['input_grad'],  rtol=1e-4, atol=1e-5)
assert torch.allclose(fsdp_weight_grad, ref['weight_grad'], rtol=1e-4, atol=1e-5)

Step 4 — Memory check

torch.cuda.reset_peak_memory_stats()
# ... run forward + backward ...
peak = torch.cuda.max_memory_allocated() / 1e9
print(f"Rank {dist.get_rank()}: peak {peak:.2f} GB")

Expected: per-rank peak memory should be ~half the single-GPU peak (since the weight is sharded across 2 GPUs). If it matches the single-GPU peak, FSDP2 is being defeated — the full weight is staying resident due to save_for_backward.

5

The Four Specific Checks — pass iff all four pass

# Check How Pass criterion
C1 Doesn't crash torchrun exits 0 No exceptions, no NCCL errors, no CUDA illegal-memory-access
C2 Forward output matches allclose(fsdp_output, ref_output) rtol=1e-4, atol=1e-5
C3 Gradients match allclose on input_grad AND weight_grad (after all-gather) rtol=1e-4, atol=1e-5
C4 Memory is sharded max_memory_allocated() per rank Per-rank peak ≤ 60% of single-GPU peak

C1 is the floor (basic compatibility). C2+C3 are correctness. C4 is the “is FSDP2 actually doing its job, or are we silently defeating it” check.

6

Possible Outcomes & Recommended Actions

Every realistic outcome falls into one of six buckets. Each has a defined next action so P8 is not stuck deciding mid-test.

A All four checks pass ✓
Success

Meaning: The autograd.Function pattern survives FSDP2. The architectural bet is validated.

Action
  • Write a 1-page result note: PyTorch version, FSDP2 API used (fully_shard vs older FSDP), exact tolerances achieved.
  • Tag the commit as fsdp2-smoke-passed.
  • Mark H15 complete. Update FORGE_V1_TARGETS.md to note FSDP2 risk retired.
  • Do not generalize to other kernels yet — that's Phase 3 work. RoPE's precomputed cos/sin tables may still surface kernel-specific issues.
B C1 fails — it crashes ×
Failure

Four likely sub-cases, each with a time-boxed fix.

Sub-cases
  • B.1 — Crash on FSDP2 wrap. Likely PyTorch too old, or weight stored as a buffer instead of nn.Parameter. Verify torch.__version__ ≥ 2.4; check list(model.norm.parameters()). Time-box 30 min.
  • B.2 — Crash during forward. Likely kernel reads weight's data pointer but FSDP2 detached storage post-all-gather. Print weight.is_contiguous(), weight.data_ptr(), weight.shape at kernel entry. Fix: weight = weight.contiguous() before passing to Triton. Time-box 45 min.
  • B.3 — Crash during backward. Likely ctx.saved_tensors returns a tensor whose storage was freed when FSDP2 discarded the full weight. Fix: save the module reference (ctx.module = self) and read self.weight in backward — FSDP2 will all-gather it again automatically. Time-box 60 min.
  • B.4 — NCCL deadlock / hang. Likely control-flow divergence between ranks. Add dist.barrier() after each major step; print rank ID at each step. Likely a bug in the test script, not the kernel. Time-box 30 min.
C C2 fails — forward output mismatches ×
Failure

Meaning: Different outputs on FSDP2 vs single-GPU. The worst kind of failure — silent numerical corruption.

Most likely causes
  • Weight read from a stale shard (kernel sees only the local shard, not the all-gathered full weight)
  • Mixed precision mismatch (FSDP2 auto-casts to bf16; single-GPU run was fp32)
  • Different RNG state across ranks for the input
Action — time-box 90 min, then escalate to Xhitij
  • Verify input x is identical on both ranks: dist.all_reduce(x.sum()) should equal 2 * ref_input.sum().
  • Print weight.shape inside the kernel — it MUST be the full [hidden] shape, not the shard. If it's the shard, FSDP2's all-gather hook didn't fire.
  • Disable mixed precision (mp_policy=None in fully_shard) and re-run. If C2 passes without MP, the issue is precision casting.
D C3 fails — gradients mismatch, forward correct ×
Failure

Meaning: Forward pattern works, backward pattern doesn't. Almost certainly an autograd.Function.backward issue.

Most likely causes
  • ctx.save_for_backward(weight, ...) saves the full all-gathered weight, but FSDP2 freed it post-forward — ctx.saved_tensors returns a corrupted reference.
  • The backward kernel writes gradients to a tensor FSDP2 doesn't know to reduce-scatter.
Action — time-box 90 min
  • Switch saving strategy — don't pass weight to autograd.Function at all. Compute the weighted output inside the nn.Module forward, not inside autograd.Function. Re-fetch weight in backward via the module.
  • If still failing, the fix is more invasive — the kernel API takes a module reference instead of raw tensors. Escalate to Xhitij with the proposed redesign before implementing.
E C4 fails — correct, but memory not sharded ▵
Partial

Meaning: Mathematically correct, but each GPU holds the full weight — FSDP2's memory savings are defeated. The most subtle failure mode and the most important one to catch.

Most likely cause
  • ctx.save_for_backward(weight) holds a reference to the full all-gathered weight, preventing FSDP2 from freeing it post-forward.
Action — time-box 60 min during hackathon, soft fail
  • Confirm with profiling — add torch.cuda.memory._snapshot() after forward and backward. The full weight should be visible if it's pinned.
  • Don't save the weight in save_for_backward. Either recompute the weight contribution in backward (FSDP2 will all-gather again, like activation checkpointing), OR save a view into the shard (advanced).
  • V1 still ships: correctness is there; memory story is weak. Fix required pre-V1, not pre-hackathon-end. Continue to V1 plan even if unresolved.
F Mixed results across runs — flaky ▵
Flaky

Meaning: First run passes, second run fails — or vice versa.

Most likely causes
  • NCCL timing race
  • Uninitialized memory in a Triton kernel (a tl.load reading undefined memory)
  • CUDA stream synchronization issue
Action — time-box 60 min
  • Run 10 times in a row. Record pass/fail rate.
  • >50% pass: likely race — add torch.cuda.synchronize() and dist.barrier() at boundaries.
  • <50% pass: likely uninitialized memory — review the Triton kernel for tl.zeros that should be initialized, or tl.load with a mask missing an other= default.
  • Flakiness is acceptable to defer if rare (<10% failure rate); document and proceed.

Decision Tree — what to do at 16:00

By 16:00 P8 should be in exactly one of these states. The hackathon's overall success does NOT depend on the smoke test passing — it depends on knowing the answer.

State at 16:00 Hackathon verdict Next step (post-hackathon)
Outcome A (all green) Success — FSDP2 risk retired Proceed to Phase 3 (FSDP2 integration) per V1 plan
Outcome B/C/D fixed within time-box Success — pattern validated with one tweak Apply same tweak to all other kernels (Day 3+)
Outcome E (memory issue, correctness OK) Partial success Memory fix becomes a tracked Phase 3 task; V1 plan unchanged
B/C/D not fixed within time-box Failure — but valuable failure Phase 3 plan needs redesign; 1-day pattern-research spike before continuing kernel work
No data (crashed cloud, time ran out) Inconclusive Re-run on Day 3 morning; do NOT extend hackathon

Anti-goals — things P8 should explicitly NOT do

Don't Wrap a full Qwen3 model

The smoke test is RMSNorm + Linear only.

Don't Add LoRA

That's Phase 3.

Don't Test multiple kernels

One kernel, one pattern question.

Don't Optimize

Correctness only. Performance is Phase 4.

Don't Exceed time-boxes while debugging

Escalate to Xhitij instead.

Don't Rewrite the kernel's Triton code

If something fails, the fix is at the autograd.Function layer, not the Triton layer (Triton kernels have no awareness of distributed at all).

Don't Burn cloud credits past 16:00

Tear down the instance, even with inconclusive results. Inconclusive + alive credits > conclusive + drained credits.
